# Архитектура UI

## Обзор

Три Angular-приложения в одном монорепо (Angular Workspace): клиент авторизации, основное приложение трекера и публичный сайт документации. Все являются клиентами поверх публичного API — никаких привилегий и прямого доступа к БД нет.

## Стек

| Слой | Технология |
|------|-----------|
| Фреймворк | Angular (latest stable) |
| Язык | TypeScript |
| Сборка | Angular CLI + esbuild |
| Стили | SCSS + CSS custom properties (дизайн-токены) |
| API-клиент | OpenAPI codegen + Zod (runtime validation) |
| Markdown-редактор | Milkdown (WYSIWYG, ProseMirror) |
| Real-time | WebSocket (events-service) |
| Контейнеризация | Docker + Nginx |

## Структура монорепо

```
frontend/
  projects/
    auth/          # клиент авторизации
    app/           # основное приложение трекера
    docs/          # публичный сайт документации
  libs/
    design-system/ # токены, компоненты, темы
    api-client/    # сгенерированный клиент + Zod-схемы
    shared-widgets/# переиспользуемые UI-компоненты (не дизайн-система)
    shared-utils/  # утилиты, хелперы
    rsql/          # построитель и парсер RSQL-выражений
    markdown-editor/ # обёртка над Milkdown
```

## Приложение: auth

Легковесный клиент авторизации — единая точка входа для регистрации, логина и подтверждения email всех клиентов системы. Реализует UI-часть OAuth 2.0 Authorization Code Flow.

### Маршрутизация

```
/login                    # форма логина, принимает client_id + redirect_uri
/register                 # форма регистрации
/verify-email             # подтверждение email по ссылке из письма → автоматический редирект в app
```

### Поведение

- Принимает параметры `client_id` и `redirect_uri` при переходе от клиента.
- После успешного логина/регистрации выполняет редирект на `redirect_uri` с authorization code.
- Подтверждение email: обрабатывает ссылку из письма, вызывает API подтверждения, автоматически перенаправляет пользователя в веб-приложение.
- Поддерживает redirect_uri на внешние адреса (веб-приложение) и localhost (CLI).

## Приложение: app

### Маршрутизация

```
/tasks                    # список задач
/tasks/:id                # просмотр/редактирование задачи
/projects                 # список проектов
/projects/:id             # проект → список задач проекта
/projects/:id/members     # участники проекта
/projects/:id/automations # автоматизации проекта (PostMVP)
/projects/:id/settings    # настройки проекта
/teams                    # список команд
/teams/:id                # команда → участники
/teams/:id/settings       # настройки команды
/invitations              # входящие и исходящие приглашения
/events                   # event feed
/tariff                   # тариф, лимиты, заморозка
/profile                  # профиль, PAT, пароль
/profile/managed          # managed-пользователи (PostMVP)
```

### UI-паттерны

**Страницы (Pages)** — полные маршруты с breadcrumbs и кнопками действий справа в заголовке.

**Модальные окна (Modals)** — не меняют URL. Используются для создания/редактирования сущностей, подтверждений, форм приглашений.

**Попапы (Popups)** — небольшие оверлеи для inline-действий (смена статуса, быстрое добавление тега, контекстное меню).

**Сайдбар** — навигация, коллапсируется на мобильном. Содержит: Tasks, Projects, Teams, Events, Docs (ссылка), счётчик входящих приглашений, переключатель темы, языка, тариф, профиль.

**Breadcrumbs** — на всех страницах. Кнопки действий контекстно справа.

### Фильтрация задач

На списке задач — два режима:
- **Визуальный конструктор:** выбор поля + оператора + значения через UI-контролы.
- **Ручной RSQL:** текстовое поле с валидацией через `libs/rsql`.

Оба режима синхронизированы — визуальный конструктор читает/пишет то же RSQL-выражение.

Литерал `me` в полях `assignee` и `author` разрешается на клиенте до отправки запроса (подставляется email текущего пользователя).

### Аутентификация и multi-auth

- Аутентификация через OAuth 2.0 Authorization Code Flow: app редиректит на клиент авторизации (auth), после логина получает authorization code, обменивает на access + refresh token через identity-service.
- Soft-refresh: при истечении access token — прозрачное обновление через refresh token без редиректа.
- **Multi-auth:** несколько аккаунтов одновременно. Данные каждого аккаунта (токены, профиль) хранятся в `localStorage` с ключом по user ID. Переключение через дропдаун в сайдбаре — мгновенное, без перезагрузки страницы.

### Real-time

WebSocket-соединение с events-service. Используется для:
- Обновления счётчика входящих приглашений в реальном времени.

Соединение открывается после логина, закрывается при логауте. При переключении аккаунта — переподключение с новым токеном.

### Markdown-редактор

Milkdown (WYSIWYG поверх ProseMirror) для поля `description` задачи. Поддерживает стандартный Markdown (заголовки, списки, код, ссылки). Хранит и отдаёт plain Markdown.

## Приложение: docs

Публичный сайт документации. Доступен без аутентификации.

**Содержимое:**
- Справочник API (генерируется из `contracts/openapi/openapi.yaml`)
- Каталог событий системы
- Справочник RSQL с примерами
- CLI-шаблоны команд для типовых операций

**Генерация:** при каждом деплое openapi.yaml перегенерируется из кода сервисов. Сайт документации пересобирается автоматически.

## Дизайн-система (libs/design-system)

Трёхуровневые токены:
1. **Global tokens** — примитивные значения (цвета, размеры, отступы).
2. **Semantic tokens** — смысловые переменные (`--color-background`, `--color-text-primary`). Переопределяются при смене темы.
3. **Component tokens** — токены конкретных компонентов (`--button-primary-bg`).

Темы (light/dark) реализованы через CSS custom properties на `:root`. Переключение — одним изменением атрибута `data-theme` на `<html>`.

## API-клиент (libs/api-client)

- Генерируется из `openapi.yaml` при сборке.
- Каждый endpoint — типизированная функция с Zod-валидацией ответа в runtime.
- При ошибке валидации — Sentry-алерт, чтобы отловить рассинхронизацию клиента и сервера.

## CI/CD

Аналогично бэкенду: GitHub Actions, path-based триггеры для `frontend/**`. Сборка — `ng build`, образ — Nginx + статика, деплой через Helm.
