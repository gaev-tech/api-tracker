# UI-спецификация

Описание клиентского приложения-трекера: навигация через сайдбар, URL-вложенность с хлебными крошками, разделение на страницы/модалки/попапы, мультиавторизация. Все действия, описанные здесь, дублируют функции из API (`api-spec.md`); UI не содержит действий, отсутствующих в API.

## 1. Общие принципы

### 1.1. Сайдбар как основная навигация

Основная навигация — постоянно присутствующий сайдбар слева. **Адаптивный:** на широких экранах (≥1024px) развёрнут с подписями пунктов; на узких — автоматически сворачивается в иконки, подписи показываются тултипами при наведении.

Структура сайдбара **сверху вниз:**

1. **Лого + название продукта** — клик ведёт на главную (`/tasks`)
2. **Задачи** → `/tasks`
3. **Проекты** → `/projects`
4. **Команды** → `/teams`
5. **События** → `/events`
6. **Документация** → открывает docs-приложение в новой вкладке (`https://docs.example.com`)
7. ——————————— (визуальный разделитель)
8. **Приглашения (N)** → `/invitations`. Счётчик pending-приглашений обновляется в реальном времени через WebSocket
9. **Тема** → попап с выбором (Light / Dark / Auto)
10. **Язык** → попап с выбором (ru / en)
11. **Текущий тариф** — иконка тарифа + название (Free / Pro / Team / Enterprise), клик ведёт на `/profile/tariff`
12. **Профиль** → попап переключения пользователей (см. 1.4). Внутри попапа — ссылка на `/profile/info`

Активный пункт сайдбара подсвечивается по верхнеуровневому разделу: при нахождении на любом URL внутри `/projects/...` подсвечен пункт «Проекты», и так для остальных.

### 1.2. Хлебные крошки и заголовок страницы

Хлебные крошки располагаются в области заголовка страницы (верхняя часть основной области). **Последний сегмент крошек играет роль заголовка страницы** — отдельного «названия» нет. Все предыдущие сегменты — кликабельные ссылки на соответствующие уровни.

**Справа от хлебных крошек** — кнопки действий текущей страницы (Создать / Импортировать / Удалить и т.п.). Все страницы следуют этому правилу.

Примеры:
- `/projects` → крошки: «Проекты»
- `/projects/:id/members` → крошки: «Проекты › Название проекта › Участники»
- `/projects/:id/automations/:aid` → крошки: «Проекты › Название проекта › Автоматизации › Название автоматизации»

### 1.3. Страницы vs модалки vs попапы

**Страницы** — полноценные URL-маршруты с хлебными крошками. Используются для всех списков, детальных view и сложных форм (задача, проект, команда, автоматизация, PAT, секрет).

**Модалки** — модальные окна поверх страницы, не меняют URL. Используются для:
- Приглашения пользователя/команды в проект/команду.
- «Шарить задачу» — выдача прямого доступа.
- Создание managed-пользователя.
- Смена пароля.
- Редактирование прав участника проекта.
- Редактирование прав прямого доступа на задачу.
- Передача владения проектом/командой.

**Попапы** — маленькие UI-слои рядом с кнопкой-инициатором, закрываются при клике вне. Используются для:
- Переключение пользователей (в сайдбаре).
- Выбор темы / языка (в сайдбаре).
- Подтверждение деструктивных действий (рядом с кнопкой).
- Автокомплит при поиске пользователя / команды.
- Добавление тега в задачу.
- Добавление проекта в задачу.
- Выбор главного проекта при создании задачи.
- Выбор ответственного (assignee).

**Нет вложенных модалок.** Модалка не открывается поверх другой модалки. Все сложные формы — страницы.

### 1.4. Мультиавторизация

Пользователь может одновременно быть авторизован под несколькими аккаунтами (например, свой аккаунт + managed-пользователи).

**Хранение:** `localStorage` содержит структуру:

```json
{
  "accounts": [
    {
      "user_id": "uuid",
      "email": "user@company.com",
      "display_name": "Иван Иванов",
      "parent_user_id": "uuid-or-null",
      "token": "pat_...",
      "added_at": "ISO 8601"
    }
  ],
  "active_user_id": "uuid"
}
```

**Попап переключения пользователей** открывается при клике на пункт «Профиль» в сайдбаре. Структура (сверху вниз):
- **Текущий активный аккаунт** — email, display_name, иконка; пометка «Текущий». Клик ведёт на `/profile/info`.
- **Другие авторизованные аккаунты** — email, display_name; клик меняет `active_user_id`, UI делает soft-refresh.
- Разделитель.
- **«Войти в другой аккаунт»** → ведёт на `/login`; после успешного логина новый аккаунт добавляется в `accounts`, становится активным.
- **«Выйти из текущего аккаунта»** — удаляет только активный; если ещё есть аккаунты — переключается на первый из оставшихся; если не было — редирект на `/login`. Вызывает `POST /auth/logout` для отзыва PAT.
- **«Выйти из всех аккаунтов»** — очищает `localStorage`, отзывает все PAT, редирект на `/login`.

**Ограничений на комбинации нет:** родитель и его managed могут быть одновременно авторизованы.

### 1.5. Real-time обновления

Приложение устанавливает одно WebSocket-подключение к `wss://api.example.com/v1/events/stream?token=<активный PAT>`. Основное использование — получение актуального счётчика pending-приглашений для сайдбара.

При переключении аккаунта соединение закрывается, открывается новое с токеном нового активного аккаунта.

Потеря соединения — автоматическое переподключение с экспоненциальной задержкой (1с → 2с → 4с → max 30с). До момента переподключения счётчик в сайдбаре показывает последнее полученное значение.

### 1.6. Тема и язык

**Тема:** Light / Dark / Auto (следует системной настройке через CSS `prefers-color-scheme`). Хранится в `/users/me` (поле `theme`).

**Язык:** ru / en. Хранится в `/users/me` (поле `language`).

Выбор через попапы в сайдбаре. На стороне клиента также кешируется в localStorage для быстрого применения до получения ответа API при старте.

## 3. Страницы аутентификации (вне сайдбара)

### 3.1. `/login`

Поля «Email» и «Пароль», кнопка «Войти», ссылка «Забыли пароль?» → `/password-reset`, ссылка «Зарегистрироваться» → `/register`.

После успешного логина — новый аккаунт добавляется в `localStorage.accounts`, становится активным, редирект на `/tasks`.

**Функции API:** Логин.

### 3.2. `/register`

Поля «Email», «Пароль», «Повторите пароль», кнопка «Зарегистрироваться», ссылка «Уже есть аккаунт? Войти».

После регистрации — сообщение о необходимости подтвердить email, редирект на `/login`.

**Функции API:** Регистрация.

### 3.3. `/password-reset`

Поле «Email», кнопка «Отправить ссылку». После отправки — информационное сообщение.

**Функции API:** Сброс пароля — запрос.

### 3.4. `/password-reset/confirm`

Открывается по ссылке из email с токеном в query-параметре. Поля «Новый пароль», «Повторите», кнопка «Сохранить». После — редирект на `/login`.

**Функции API:** Сброс пароля — подтверждение.

### 3.5. `/email-verify`

Открывается по ссылке из email с токеном в query-параметре. При валидном токене — сообщение «Email подтверждён» + кнопка «Войти» → `/login`. При невалидном/истёкшем — сообщение об ошибке.

**Функции API:** Подтверждение email.

## 4. Задачи

### 4.1. `/tasks` — список задач

**Крошки:** «Задачи». **Справа:** кнопка «Создать задачу» → `/tasks/new`.

**Под крошками:**
- **В штатном режиме** — панель фильтра: виджет RSQL-фильтра (виджет в разделе 9.3), счётчик «Найдено: N задач».
- **При выделении ≥1 задачи** чекбоксами — панель фильтра заменяется на **панель массовых действий** (счётчик «Выбрано: N», кнопка выбора операции с попапом: изменить статус / назначить assignee / добавить теги / удалить теги / добавить блокеры / убрать блокеры / прикрепить к проекту / открепить от проекта / выдать прямой доступ / отозвать прямой доступ / удалить). При снятии всех чекбоксов — возвращается панель фильтра.

**Таблица задач:**
- Чекбокс выделения; в шапке — чекбокс «Все», выделяющий **все задачи под текущим фильтром** (не только на странице).
- Колонки: `id` (сокращённый, копируемый), `title` (клик ведёт на `/tasks/:id/info`), `status` (бейдж), `assignee` (email + avatar), `projects` (чипы), `tags` (чипы), `updated_at`.
- Курсорная пагинация.
- Сортировка по `created_at` / `updated_at` / `title`.

**Логика «Все» с частичными снятиями.** При активном режиме «Все» пользователь может снять отдельные задачи чекбоксами — UI запоминает exclude-список на клиенте. Массовая операция применится ко всем задачам под фильтром кроме исключённых.

**Массовая операция при выборке >1000 задач:**
1. UI вызывает `GET /tasks/ids?filter=<RSQL>` постранично (по 1000 ID за страницу), собирает все ID.
2. Вычитает exclude-список.
3. Вызывает `POST /tasks/bulk/by-ids` пачками по 1000 ID.
4. Отображает прогресс-бар «Обработка задач: X из Y».
5. Транзакционность — на уровне пачки: ошибка на пачке K останавливает операцию, пачки 1…K-1 уже применены. Пользователь видит уведомление с конкретными числами.

**Функции API:** Получить список задач, Массовое действие по RSQL, Массовое действие по списку ID, Получить массив ID под фильтром.

### 4.2. `/tasks/new` — создание задачи

**Крошки:** «Задачи › Создать задачу». **Справа:** «Создать», «Отмена».

**Форма:**
- Поле «Название» (обязательное).
- Поле «Описание» (Milkdown-редактор).
- Селектор «Статус» (default `opened`).
- Кнопка «Ответственный» — **попап выбора ответственного** (автокомплит).
- Кнопка «Главный проект» — **попап выбора главного проекта** (один из доступных пользователю проектов с R-12, или «Без проекта» для личной задачи).
- Кнопка «Добавить в другие проекты» — **попап добавления проекта**.
- Кнопка «Добавить тег» — **попап добавления тега**.
- Кнопка «Добавить блокер» — автокомплит задач.

**Функции API:** Создать задачу.

После успешного создания — редирект на `/tasks/:id/info`.

### 4.3. `/tasks/:id/info` — редактирование задачи

**Крошки:** «Задачи › <Название задачи> › Информация». **Справа:** кнопка «Удалить задачу» (если R-8) → попап подтверждения.

**Форма:**
- `title` — inline-редактирование (при R-1).
- `description` — Milkdown (при R-2). Drop файлов в редактор → автозагрузка в `/files` с `task_id` → вставка ссылки в markdown.
- `status` — селектор (при R-6).
- Блок «Ответственный» — текущий assignee с крестиком (при R-5); кнопка «+» — попап выбора пользователя.
- Блок «Теги» — чипы с крестиками (при R-3); кнопка «+» — попап тега.
- Блок «Проекты» — чипы с крестиками (при R-12 в соответствующем проекте); кнопка «+» — попап добавления проекта.
- Блок «Блокеры» — список с крестиками (при R-4); кнопка «+» — автокомплит задач. Read-only: список задач, которые блокирует данная.
- Блок «Мета»: автор, дата создания, дата последнего изменения.

**Функции API:** Получить задачу по ID, Изменить название/описание/статус/assignee/теги/блокеры, Прикрепить/открепить от проекта, Удалить задачу.

### 4.4. `/tasks/:id/accesses` — прямые доступы задачи

**Крошки:** «Задачи › <Название задачи> › Доступы». **Справа:** «Шарить задачу» — **модалка выдачи доступа**. Видно только при R-7.

**Таблица прямых доступов:**
- Получатель (email пользователя или название команды, с иконкой типа).
- Чипы прав R-1…R-8.
- Автор выдачи.
- Дата.
- Кнопки «Изменить» (модалка редактирования), «Отозвать» (попап подтверждения).

**Модалка «Шарить задачу»:**
- Попап автокомплита пользователя / команды.
- Виджет выбора прав R-1…R-8 (с маской).
- Кнопка «Выдать».

**Модалка «Редактировать права прямого доступа»:**
- Получатель (read-only).
- Виджет выбора прав R-1…R-8.
- Кнопка «Сохранить».

**Функции API:** Получить список прямых доступов, Выдать / изменить / отозвать прямой доступ.

## 5. Проекты

### 5.1. `/projects` — список

**Крошки:** «Проекты». **Справа:** «Создать проект» → `/projects/new`.

**Фильтр:** поиск по названию, переключатель «Только мои владения».

**Таблица:**
- Название (клик → `/projects/:id/members`).
- Владелец.
- Моя роль — чипы прав.
- Число задач (клик → `/tasks?filter=projects==<id>`).
- Статус: «Активен» / «Заблокирован тарифом».
- Дата создания.

**Функции API:** Получить список проектов.

### 5.2. `/projects/new` — создание

**Крошки:** «Проекты › Создать проект». **Справа:** «Создать», «Отмена».

Поле «Название». Создатель автоматически получает R-1…R-14.

**Функции API:** Создать проект.

### 5.3. `/projects/:id/members` — участники

**Крошки:** «Проекты › <Название> › Участники». **Справа:** «Пригласить пользователя», «Пригласить команду» — **модалки**.

**Секции:**

**Активные пользователи-участники:**
- Email, чипы прав, дата вступления.
- Кнопки «Изменить права» (модалка), «Удалить» (попап подтверждения).
- В строчке с текущим пользователем (если владелец и pending-передача) — индикатор + «Отменить передачу».

**Активные команды-участники:**
- Название, чипы прав, статус («Активна» / «Заморожена в проекте»), дата.
- Кнопки «Изменить права», «Удалить».

**Pending-приглашения пользователям:**
- Email, предложенные права, активно / неактивно, автор, дата.
- Кнопка «Отозвать».

**Pending-приглашения командам** — аналогично.

**Модалка «Пригласить пользователя/команду»:**
- Попап автокомплита (пользователь по email или команда по названию).
- Виджет выбора прав R-1…R-14.
- Кнопка «Пригласить».

**Модалка «Редактировать права участника проекта»:**
- Участник (read-only).
- Виджет выбора прав.
- Кнопка «Сохранить».

**Функции API:** Получить участников, Пригласить, Изменить права, Удалить, Отозвать приглашение.

### 5.4. `/projects/:id/automations` — список автоматизаций

**Крошки:** «Проекты › <Название> › Автоматизации». **Справа:** «Создать автоматизацию» → `/projects/:id/automations/new`. При R-11.

**Таблица:**
- Название (клик → редактирование).
- Тип события-триггера.
- Условие (сокращённо).
- Toggle «Вкл/Выкл» (без подтверждения).
- Счётчик подряд неуспехов.
- Плашка «Авто-выключена после 10 неуспехов» (если применимо).
- Дата создания.

**Функции API:** Получить список автоматизаций, Включить/выключить.

### 5.5. `/projects/:id/automations/new` — создание

**Крошки:** «Проекты › <Название> › Автоматизации › Создать автоматизацию».

Форма — такая же, как на странице редактирования (5.6), без блока истории срабатываний.

**Функции API:** Создать автоматизацию.

### 5.6. `/projects/:id/automations/:aid` — редактирование

**Крошки:** «Проекты › <Название> › Автоматизации › <Название автоматизации>». **Справа:** «Сохранить», «Удалить» (попап подтверждения), toggle «Вкл/Выкл».

**Форма (верх страницы):**

*Триггер:* селектор «Тип события» (сгруппированы по сущности).

*Условие:* виджет RSQL-фильтра (опционально).

*Действие:* селектор HTTP-метода, поле «URL» (с плейсхолдерами `{{…}}` и `{{secrets.*}}`), таблица заголовков, поле «Тело запроса». Блок помощи: список доступных плейсхолдеров + список секретов проекта.

*Тестирование:* кнопка «Выполнить тестовый вызов».

**История срабатываний (внизу страницы):**
- Таблица: время, номер попытки, статус, HTTP-код, выдержка из ответа, ссылка на событие.
- Фильтры (статус, период), пагинация.

**Функции API:** Получить автоматизацию, Изменить, Удалить, Включить/выключить, Получить историю.

### 5.7. `/projects/:id/secrets` — список секретов

**Крошки:** «Проекты › <Название> › Секреты». **Справа:** «Создать секрет» → `/projects/:id/secrets/new`. При R-11.

Таблица: ключ (без значения), даты, кнопки «Изменить» → `/projects/:id/secrets/:key`, «Удалить» (попап подтверждения).

**Функции API:** Получить список секретов.

### 5.8. `/projects/:id/secrets/new` и `/projects/:id/secrets/:key` — создание/редактирование

**Крошки:** «Проекты › <Название> › Секреты › <Ключ / Создать секрет>».

Форма: поле «Ключ» (при редактировании read-only), поле «Значение» (текущее значение не возвращается).

**Функции API:** Создать/обновить/удалить секрет.

### 5.9. `/projects/:id/import` — CSV-импорт

**Крошки:** «Проекты › <Название> › Импорт». При R-14.

**Содержимое:**

*Форма импорта (сверху):*
- Область загрузки CSV (drop zone + кнопка выбора).
- Блок описания формата: обязательная колонка `title`, опциональные `description`/`status`/`assignee`/`tags`; разделитель тегов `;`; лимит 10 000 строк / 10 МБ.
- Кнопка «Импортировать».

*Результат импорта (появляется снизу после нажатия):*
- При успехе — «N задач импортировано», ссылка на фильтр задач проекта.
- При ошибке — «Импорт отклонён из-за ошибок валидации». Таблица ошибок: номер строки, колонка, описание. Клик по строке раскрывает детали.

История импортов не сохраняется. Пользователь может исправить CSV и загрузить снова — результаты обновятся.

**Функции API:** Импортировать CSV.

### 5.10. `/projects/:id/settings` — настройки

**Крошки:** «Проекты › <Название> › Настройки».

**Блоки:**
- **Название** — inline-редактирование (при R-9).
- **Владелец** — кнопка «Передать владение» (**модалка**). Индикатор pending-передачи + «Отменить».
- **Опасная зона:** «Удалить проект» (при R-13) — попап подтверждения.

**Модалка «Передать владение проектом»:**
- Попап автокомплита пользователей.
- Подтверждение.

**Функции API:** Переименовать, Передать владение, Удалить проект.

## 7. События, Приглашения, Тарифы

### 7.1. `/events` — лента событий

**Крошки:** «События».

**Боковая панель фильтра:**
- Автор — автокомплит (мультивыбор).
- Проект — мультивыбор.
- Задача — автокомплит.
- Тип события — мультиселект (группировка по сущности).
- Период — date range picker.
- Кнопки «Применить», «Сбросить».

**Лента:**
- Время, тип события (бейдж), автор, контекст (ссылки), краткое описание. Клик раскрывает JSON payload.
- Курсорная пагинация.

**Функции API:** Получить поток событий.

### 7.2. `/invitations` — входящие приглашения

**Крошки:** «Приглашения».

Только **входящие** приглашения текущему пользователю. Исходящие — в соответствующих местах (модалки проектов/команд, строчках участников). Счётчик в сайдбаре — через WebSocket.

**Секции:**
- Приглашения в команды — название, владелец, автор, дата; «Принять» / «Отклонить».
- Приглашения в проекты — название, владелец, предложенные права, автор, дата; «Принять» / «Отклонить». Для приглашения команды, где я админ — «Команда X приглашена в проект Y».
- Передачи владения мне — проект/команда, текущий владелец, дата; «Принять» / «Отклонить».

**Функции API:** Принять/отклонить приглашение в проект, Принять/отклонить приглашение в команду, Принять/отклонить передачу владения проектом, Принять/отклонить передачу владения командой.

### 7.3. `/tariffs` — тарифы

**Крошки:** «Тарифы».

**Сводка текущего тарифа.**

**Таблица тарифов** (Free / Pro / Team / Enterprise): лимиты (задачи, проекты, команды, автоматизации, участники, managed, хранилище), цены (monthly/yearly). Кнопки:
- Текущий: «Текущий».
- Апгрейд: «Перейти» → модалка выбора периода → ЮKassa.
- Даунгрейд: «Запланировать» → попап подтверждения с пояснением «Текущий тариф будет действовать до <дата>, затем активируется выбранный».

**Функции API:** Получить список тарифов, Инициировать апгрейд, Инициировать даунгрейд.

## 8. Профиль

### 8.1. `/profile/info` — личные данные

**Крошки:** «Профиль › Информация».

**Блоки:**
- Email (read-only).
- Display name (редактируется).
- Смена темы/языка (дублирование попапов для удобства).
- Кнопка «Сменить пароль» — **модалка** с полями «Текущий пароль», «Новый пароль», «Повторите».

**Опасная зона:**
- Кнопка «Удалить аккаунт» — **модалка подтверждения** с детальным предупреждением:
  - «Все ваши проекты (N), команды (M), managed-пользователи (K), автоматизации, секреты, PAT будут удалены безвозвратно.»
  - «Все задачи, где вы указаны как автор или исполнитель, сохранятся, но ссылки на вас станут "удалённый пользователь".»
  - Поле ввода «Для подтверждения введите свой email».
  - Кнопка «Удалить навсегда».

**Функции API:** Получить профиль, Изменить профиль, Смена пароля, Удалить свой аккаунт.

### 8.2. `/profile/pats` — список PAT

**Крошки:** «Профиль › PAT». **Справа:** «Создать PAT» — **модалка**.

**Таблица:**
- Имя.
- Значение токена (полное, копируемое).
- Дата создания, срок истечения, статус.
- Кнопка «Отозвать» (попап подтверждения).

**Модалка «Создать PAT»:** поле «Имя», поле «Срок истечения» (опц.), кнопка «Создать». После создания — токен виден в таблице.

**Функции API:** Получить список PAT, Создать PAT, Отозвать PAT.

### 8.3. `/profile/managed` — managed-пользователи

**Крошки:** «Профиль › Managed-пользователи». Видно только обычным пользователям. **Справа:** «Создать managed» — **модалка**.

Индикатор: «N managed-пользователей из M по тарифу».

**Таблица:**
- Email.
- Статус: «Ожидает подтверждения email» / «Активен» / «Деактивирован».
- Дата создания.
- Действия в строчке (зависят от статуса):
  - `pending_email_verification` — «Переотправить письмо», «Удалить» (попап).
  - `active` — toggle активности (деактивирует без подтверждения), «Сбросить пароль» (модалка), «Удалить» (попап).
  - `deactivated` — toggle (ре-активирует), «Сбросить пароль», «Удалить».

**Модалка «Создать managed-пользователя»:** поле «Email» (уникальный), поле «Пароль», кнопка «Создать». После — пояснение об отправке письма подтверждения.

**Модалка «Сбросить пароль managed-пользователя»:** email (read-only), поле «Новый пароль», поле «Повторите», кнопка «Сохранить».

**Для managed-пользователя** этот раздел заменяется блоком: «Ваш аккаунт управляется пользователем <email родителя>. Вы не можете менять тариф, передавать владение или менять email.»

**Функции API:** Получить managed, Создать, Деактивировать, Ре-активировать, Сбросить пароль, Удалить, Переотправить письмо.

### 8.4. `/profile/tariff` — тариф и подписка

**Крошки:** «Профиль › Тариф».

**Блоки на одной странице:**
- Текущий тариф, период, дата окончания.
- Банк дней — таблица слоёв.
- Запланированный даунгрейд (если есть) — плашка + «Отменить даунгрейд».
- Ссылка «Сменить тариф» → `/tariffs`.
- Использование лимитов — прогресс-бары (задачи, проекты, команды, автоматизации, участники, managed, хранилище).
- История смен тарифа — таблица (коллапс-блок).
- Ссылка на «Заблокированные сущности» (если есть).
- Для Enterprise — ссылка «Управление слотами» → `/profile/tariff/slots`.

**Функции API:** Получить подписку, Получить историю, Получить использование, Отменить запланированный даунгрейд.

### 8.5. `/profile/tariff/slots` — слоты Enterprise

**Крошки:** «Профиль › Тариф › Слоты». Только на Enterprise.

- Текущее число слотов.
- Контрол «+/−».
- При увеличении — «Купить» → ЮKassa.
- При уменьшении — «Запланировать уменьшение» → попап подтверждения. Если уменьшение запланировано — «Отменить уменьшение».

**Функции API:** Купить слоты, Запланировать уменьшение, Отменить уменьшение.

### 8.6. `/profile/blocked` — заблокированные сущности

**Крошки:** «Профиль › Заблокированные».

Вводный блок: текущий тариф, лимиты, пояснение блокировок.

**Секции:**
- Заблокированные задачи.
- Заблокированные проекты.
- Заблокированные команды.
- Заблокированные автоматизации.
- Заблокированные участники проектов.
- Заблокированные участники команд.

Для каждой секции — пояснение «Разблокируется автоматически, когда…».

**Функции API:** Получить список заблокированных сущностей.

## 9. Переиспользуемые UI-элементы

### 9.1. Попап переключения пользователей

Описан в разделе 1.4.

### 9.2. Попап темы и языка

- Тема: Light / Dark / Auto.
- Язык: ru / en.
- Клик по опции сразу применяет изменение, попап закрывается, изменение уходит в `/users/me`.

### 9.3. Виджет RSQL-фильтра

**Где используется:** `/tasks`, `/events`, `/projects/:id/automations/:aid` (поле условия).

**Структура:**
- Текстовое поле с подсветкой синтаксиса.
- Автодополнение: имена полей, операторы, значения enum (`status`), литерал `null`, относительные даты (`now-7d`).
- Inline-ошибки.
- Счётчик длины (лимит 4096).
- Кнопка «Помощь» → попап с примерами.

### 9.4. Виджет выбора прав

**Где используется:** модалки выдачи/изменения прямого доступа (R-1…R-8), модалки приглашения/редактирования прав в проекте (R-1…R-14).

**Структура:**
- Группы чекбоксов: «Права на задачи» (R-1…R-8) и «Права на проект» (R-9…R-14).
- Каждое право — чекбокс с подписью и тултипом.
- При выдаче другим — чекбоксы недоступных вам прав отключены (R-7 ограничение).
- Пресеты: «Полные права», «Только чтение», «Снять все».

### 9.5. Попап автокомплита пользователя/команды

**Где используется:** приглашения, шаринг задачи, выбор assignee, передача владения, фильтры событий.

**Структура:**
- Поле ввода с debounce 300мс, минимум 2 символа.
- Результаты: для пользователей — email + display_name + аватар; для команд — название + владелец.
- Одиночный или мультивыбор (в зависимости от контекста).

### 9.6. Попап подтверждения деструктивного действия

**Где используется:** удаление задачи, проекта, команды, автоматизации, секрета, участника, PAT, managed; отзыв прямого доступа; отзыв приглашения; отмена передачи владения.

**Структура:**
- Текст с последствиями.
- Кнопка «Подтвердить» (destructive-стиль).
- Кнопка «Отмена».
- Открывается рядом с кнопкой-инициатором.
- Для особо критичных действий (удаление аккаунта, проекта с задачами) — полноценная модалка с вводом подтверждающего текста.

### 9.7. Markdown-редактор (Milkdown)

**Где используется:** поле `description` задачи.

- WYSIWYG-редактирование.
- Drop файлов → загрузка через `POST /files` и вставка ссылки в markdown.
- Read-only режим.
