# Облачный трекер задач — продуктовая спецификация

# Описание

## Проблема

Современные системы управления задачами проектируются вокруг пользовательского интерфейса, а не программной модели. В результате API, автоматизации и интеграции оказываются вторичными и наследуют ограничения UI вместо того, чтобы отражать стабильную доменную логику.

Это делает внешнее взаимодействие с системой фрагментированным и плохо предсказуемым: отсутствуют единые контракты, поведение операций зависит от скрытых внутренних правил, а ошибки и данные не всегда представлены в форме, пригодной для автоматической обработки. В итоге такие системы сложно использовать как инфраструктурный слой для CI/CD, интеграций и автоматизаций.

## Описание продукта

Облачный таск-трекер, построенный по принципу API-first, где весь API системы спроектирован для удобного использования из консоли и автоматизаций, а не как отражение UI.

Программный интерфейс является основной и полной моделью взаимодействия с системой: через него выполняются все операции, управляются данные и состояния, а UI выступает лишь одним из клиентов поверх единого доменного контракта. Это позволяет использовать систему как инфраструктурный компонент с предсказуемым поведением и консистентной моделью работы в любых сценариях интеграции.

## Ключевые возможности

### Публичное API
Широкое публичное API покрывает все операции системы и является основной моделью взаимодействия. Через API доступны все действия с задачами, проектами, командами, доступами и тарифами. API поддерживает массовые операции в двух режимах — атомарном (bulk) и частичном (batch), с выбором целевого множества по RSQL-выражению или по явному списку идентификаторов.

### CLI
Консольный клиент покрывает все операции API и предназначен для работы с системой из терминала, скриптов и CI/CD. Поддерживает интерактивный логин через браузер и неинтерактивную аутентификацию через PAT.

### Документация с CLI-шаблонами
Документация содержит готовые CLI-шаблоны для типовых операций с системой — их можно копировать и выполнять сразу, превращая справочник в инструмент прямого управления.

### Двусторонние автоматизации
Автоматизации позволяют системе быть активным участником интеграционного контура. Она может как вызывать внешние системы через HTTP-запросы, так и реагировать на внешние вызовы и события. Это делает систему связующим слоем между сервисами, а не пассивным хранилищем данных.

### RSQL как единый контракт работы с данными
RSQL используется как универсальный язык фильтрации для API-запросов, автоматизаций и массовых операций. Он задаёт единый способ описания выборок сущностей независимо от контекста использования и исключает различия между UI, API и интеграциями.

### PAT-токены и внешние подключения
PAT-токены обеспечивают стандартный способ подключения внешних систем и агентов к API. Они работают с теми же правами, что и пользователь, и обеспечивают предсказуемую модель авторизации для интеграций и автоматических процессов.

## Клиенты

### Клиент авторизации

Отдельное легковесное веб-приложение, являющееся единым authorization server системы (OAuth 2.0 Authorization Code Flow). Обслуживает все потоки аутентификации для всех клиентов:

- Регистрация нового пользователя.
- Логин — единая точка входа для веб-приложения и CLI. Клиент получает authorization code и обменивает на access + refresh token.
- Подтверждение email — пользователь переходит по ссылке из письма, клиент авторизации подтверждает email через API и автоматически перенаправляет в веб-приложение.
- Логаут.

Деплоится в одной джобе с остальными клиентами и API.

### Веб-приложение

Основной интерфейс системы. Является клиентом поверх публичного API — не имеет привилегий, прямого доступа к базам данных или внутренних эндпоинтов. Любое действие, доступное в UI, доступно и через API.

Аутентификация выполняется через клиент авторизации (OAuth 2.0 Authorization Code Flow): веб-приложение редиректит на клиент авторизации, после логина получает access + refresh token и выполняет запросы к API от имени пользователя.

### Сайт документации

Публично доступный сайт без необходимости авторизации. Содержит:

- полный справочник API с описанием всех endpoint, параметров, структур запросов и ответов, кодов ошибок;
- каталог событий системы;
- справочник RSQL с примерами;
- готовые CLI-шаблоны команд для выполнения типовых операций.

Документация генерируется из OpenAPI-спецификации и поддерживается актуальной автоматически.

### CLI-приложение

Консольный клиент для взаимодействия с API. Покрывает все операции публичного API и предназначен для использования в скриптах, CI/CD и автоматизациях.

Поддерживает два способа аутентификации:
- **Интерактивный логин** — CLI открывает браузер на клиенте авторизации с указанием идентификатора приложения и redirect_uri на локальный порт. После логина клиент авторизации редиректит обратно, CLI получает токен.
- **PAT** — для неинтерактивного использования (скрипты, CI/CD). Передаётся через флаг `--token` или переменную окружения `TRACKER_TOKEN`.

## Глоссарий

Task / Задача — единица работы в системе, представляющая задачу с описанием, статусом, участниками, связями и доступами.

Project / Проект — контейнер задач, объединяющий их в единый контекст с владельцем, участниками, правами и автоматизациями.

User / Пользователь — пользователь системы, который может быть обычным или managed и является базовой сущностью взаимодействия с платформой.

Managed User / Управляемый пользователь — пользователь, созданный и управляемый другим пользователем (родителем) с ограниченными административными возможностями.

Team / Команда — группа пользователей, объединённая в организационную единицу с общими доступами и владельцем.

Team Member / Участник команды — пользователь, состоящий в команде и обладающий правами, определяемыми доступами команды.

Team Admin / Администратор команды — участник команды с расширенными правами управления составом и состоянием команды.

Project Member / Участник проекта — пользователь или команда, добавленные в проект с набором проектных прав.

Owner / Владелец — пользователь, определяющий тариф, лимиты и управление жизненным циклом проекта или команды.

Access / Доступ — набор прав на сущность, определяющий допустимые операции для пользователя или команды.

Direct Access / Прямой доступ — доступ к конкретной задаче с набором прав, выданный напрямую пользователю или команде.

Project Access / Проектный доступ — набор прав участника проекта, распространяющийся на все задачи внутри проекта.

Permission (R-right) / Право (R-право) — атомарное разрешение на выполнение конкретной операции над задачей или проектом.

Tariff / Тариф — подписка пользователя, определяющая лимиты, доступные функции и поведение ресурсов системы.

Limit / Лимит — числовое ограничение на количество сущностей или операций в рамках тарифа.

Task Status / Статус задачи — состояние задачи (opened, progress, closed), определяющее её этап жизненного цикла.

Tag / Тег — строковая метка задачи, используемая для фильтрации и группировки.

Blocking Task / Блокирующая задача — задача, препятствующая выполнению другой задачи.

Blocked Task / Заблокированная задача — задача, зависящая от других задач и недоступная к завершению до их закрытия.

Automation / Автоматизация — правило, связывающее события системы с API-вызовами или внешними действиями.

Event / Событие — фиксируемое изменение состояния сущности, используемое для интеграций и автоматизаций.

RSQL / RSQL-запрос — единый язык фильтрации данных, используемый во всех контекстах системы.

PAT (Personal Access Token) / Персональный токен доступа — ключ доступа, позволяющий выполнять API-запросы от имени пользователя.

Auth Client / Клиент авторизации — отдельное легковесное веб-приложение, являющееся единым authorization server системы (OAuth 2.0 Authorization Code Flow) для всех клиентов.

CLI Template / CLI-шаблон команды — готовая консольная команда для выполнения операций системы через документацию.

Bulk Operation / Массовая операция (атомарная) — операция, которая либо полностью выполняется для всех сущностей, либо полностью откатывается.

Batch Operation / Массовая операция (пакетная) — операция с частичным выполнением и детальным результатом по каждой сущности.

Secret / Секрет — защищённое значение проекта, используемое в автоматизациях и недоступное в открытом виде через API.

Import / Импорт — операция массовой загрузки задач в проект из внешнего формата (например, CSV) с валидацией и транзакционностью.

Share / Шаринг (раздача доступа) — процесс выдачи прямых доступов к задаче другим пользователям или командам.

Bank Days / Банк дней — механизм накопления неиспользованных оплаченных дней тарифа для последующего применения при смене подписки.

Freeze / Заморозка (замороженная сущность) — состояние сущности, при котором она доступна только для чтения из-за тарифных ограничений, применяемых при превышении лимитов или изменении подписки.

User Deactivation / Деактивация пользователя (неактивный пользователь) — состояние пользователя, при котором он сохраняется в системе, но не может выполнять вход и действия из-за тарифных ограничений или административной деактивации до момента повторной активации.

## Модель доступов

Система использует унифицированную модель доступов, в которой все права выражаются через набор R-прав (R-1…R-14) и применяются к задачам и проектам через прямые доступы, проектные доступы и доступы команд.

## Пользователи, проекты и команды

Пользователь (User) является базовой сущностью системы и может создавать проекты и команды, а также участвовать в них.

Проект (Project) является основным контейнером задач и определяет контекст работы, в рамках которого распределяются права и доступы.

Команда (Team) представляет группу пользователей, объединённых для совместного участия в проектах и получения доступов как единая сущность.

Команда существует под управлением владельца и содержит участников и администраторов. Администраторы команды могут управлять её составом и ролями участников, а также участвуют в получении всех доступов, назначенных команде. Участники команды получают доступы, назначенные команде.

Проект имеет владельца и участников. Участниками проекта могут быть как пользователи, так и команды. Владельцы проектов определяют тарифные ограничения и жизненный цикл проекта. Владельцы команд также определяют тарифные ограничения и жизненный цикл команды.

## Типы доступов

Доступ в системе всегда представляет собой набор R-прав.

**Прямой доступ (Direct Access)** — доступ к конкретной задаче, выдаваемый пользователю или команде. Он применяется только к этой задаче и включает подмножество прав R-1…R-8.

**Проектный доступ (Project Access)** — доступ, получаемый участником проекта (пользователем или командой), который распространяется на все задачи проекта. Он включает права R-1…R-14 в рамках контекста проекта.

**Право (Permission, R-right)** — атомарная операция, определяющая возможность выполнения конкретного действия над задачей или проектом.

**Чтение** не является отдельным правом. Видимость задачи определяется контекстом: участник проекта видит все задачи проекта; наличие прямого доступа с любым правом даёт чтение конкретной задачи. Без доступа ни через проект, ни напрямую — задача невидима.

## Вычисление итоговых прав

Итоговые права пользователя на задачу формируются как объединение всех источников доступа:

* прямые доступы к задаче;
* доступы команды, в которой состоит пользователь;
* проектные доступы пользователя;
* проектные доступы команды.

Все источники суммируются по принципу объединения прав (union), без приоритета одного источника над другим.

## Шаринг и управление доступами

Шаринг задачи (R-7) позволяет выдавать прямые доступы другим пользователям и командам.

При выдаче доступа действует ограничение: пользователь не может выдать больше прав, чем имеет сам на момент выдачи.

Аналогичное ограничение применяется ко всем механизмам выдачи прав в системе, включая управление участниками проекта (R-10): пользователь не может назначить другим пользователям или командам больше прав, чем имеет сам в соответствующем контексте доступа.

## Заморозка

При превышении тарифных лимитов или изменении тарифа часть сущностей переводится в состояние Freeze — только чтение, все операции изменения блокируются до восстановления соответствия тарифу. Freeze применяется автоматически.

## Владение сущностями

Каждый проект и каждая команда имеют владельца (User), определяющего тарифные ограничения и жизненный цикл. Тариф владельца используется для расчёта лимитов. Передача владения выполняется через подтверждаемую операцию.

## Тарифная система

Тариф привязан к пользователю и определяет лимиты: задачи, проекты, команды, автоматизации, участники. Тариф владельца распространяется на все его проекты, команды и managed-пользователей — ресурсы учитываются в рамк��х одного тарифного пула. Система поддерживает четыре тарифа: Free, Pro, Team, Enterprise.

# Требования

## 1. Функциональные требования

### 1.1 Задачи

#### 1.1.1 Поля задачи

1.1.1.1. **Идентификатор задачи.** Задача имеет уникальный идентификатор (id), создаваемый системой при создании и неизменяемый.

1.1.1.2. **Название задачи.** Задача имеет название (title), редактируемое по праву R-1 согласно п. 1.2.1.1 «Редактирование названия задачи».

1.1.1.3. **Описание задачи.** Задача имеет описание (description) в формате Markdown, редактируемое по праву R-2 согласно п. 1.2.1.2 «Редактирование описания задачи».

1.1.1.4. **Статус задачи.** Задача имеет статус (status) со значениями: opened, progress, closed. Изменяется по праву R-6 согласно п. 1.2.1.6 «Изменение статуса задачи».

1.1.1.5. **Автор задачи.** Задача имеет автора (author) — пользователя, создавшего задачу. Устанавливается при создании и не изменяется.

1.1.1.6. **Исполнитель задачи.** Задача имеет исполнителя (assignee). Изменяется по праву R-5 согласно п. 1.2.1.5 «Управление исполнителем задачи».

1.1.1.7. **Проекты задачи.** Задача имеет список проектов (projects). Изменяется через прикрепление и открепление по праву R-12 согласно п. 1.2.1.11 «Прикрепление и открепление задач».

1.1.1.8. **Теги задачи.** Задача имеет массив тегов (tags), редактируемый по праву R-3 согласно п. 1.2.1.3 «Редактирование тегов задачи».

1.1.1.9. **Блокирующие задачи.** Задача имеет список задач-блокеров (blocking_tasks), редактируемый по праву R-4 согласно п. 1.2.1.4 «Редактирование блокеров задачи».

1.1.1.10. **Заблокированные задачи.** Задача имеет вычисляемый список задач (blocked_tasks), заблокированных текущей. Список формируется системой автоматически и не редактируется вручную.

1.1.1.11. **Дата создания задачи.** Задача имеет дату создания (created_at), устанавливаемую системой при создании и неизменяемую.

1.1.1.12. **Дата обновления задачи.** Задача имеет дату последнего изменения (updated_at), обновляемую системой автоматически при любых изменениях задачи.

#### 1.1.2 Жизненный цикл задачи

1.1.2.1. **Условие существования задачи.** Задача существует, пока прикреплена хотя бы к одному проекту ИЛИ имеет хотя бы один прямой доступ (п. 1.2.3 «Прямой доступ») с любым правом.

1.1.2.2. **Автоматическое удаление задачи.** Задача удаляется автоматически, если у неё не осталось ни прямых доступов, ни проектных доступов.

1.1.2.3. **Удаление задачи по праву.** Задача может быть удалена пользователем, обладающим правом R-8 согласно п. 1.2.1.8 «Удаление задачи».

#### 1.1.3 API задач

1.1.3.1. **Создание задачи.** Система предоставляет API для создания задачи.

1.1.3.2. **Получение задачи по ID.** Система предоставляет API для получения задачи по идентификатору.

1.1.3.3. **Список задач.** Система предоставляет API для получения списка задач с RSQL-фильтром (п. 1.9 «RSQL») и курсорной пагинацией с сортировкой по created_at asc.

1.1.3.4. **Литерал me в фильтрах задач.** При фильтрации списка задач поддерживается литерал `me` для полей assignee и author, подставляющий идентификатор текущего пользователя.

1.1.3.5. **Список задач проекта.** Система предоставляет API для получения списка задач проекта (`GET /projects/{id}/tasks`) с теми же фильтрами, что и в п. 1.1.3.3 «Список задач».

1.1.3.6. **Обновление задачи.** Система предоставляет API для частичного обновления задачи (PATCH): title, description, status, assignee, tags, blocking_tasks, projects.

1.1.3.7. **Удаление задачи через API.** Система предоставляет API для удаления задачи.

### 1.2 Доступы и права

#### 1.2.1 Каталог прав

1.2.1.1. **Редактирование названия задачи (R-1).** Право R-1 позволяет редактировать название задачи (title).

1.2.1.2. **Редактирование описания задачи (R-2).** Право R-2 позволяет редактировать описание задачи (description).

1.2.1.3. **Редактирование тегов задачи (R-3).** Право R-3 позволяет редактировать теги задачи (tags).

1.2.1.4. **Редактирование блокеров задачи (R-4).** Право R-4 позволяет редактировать список блокирующих задач (blocking_tasks).

1.2.1.5. **Управление исполнителем задачи (R-5).** Право R-5 позволяет изменять исполнителя задачи (assignee).

1.2.1.6. **Изменение статуса задачи (R-6).** Право R-6 позволяет изменять статус задачи (status).

1.2.1.7. **Шаринг задачи (R-7).** Право R-7 позволяет выдавать, изменять и отзывать прямые доступы к задаче с ограничением согласно п. 1.2.7.2 «Ограничение на выдачу прав».

1.2.1.8. **Удаление задачи (R-8).** Право R-8 позволяет удалять задачу.

1.2.1.9. **Переименование проекта (R-9).** Право R-9 позволяет переименовывать проект.

1.2.1.10. **Управление участниками проекта (R-10).** Право R-10 позволяет добавлять и удалять пользователей и команды из проекта с ограничением согласно п. 1.2.7.2 «Ограничение на выдачу прав».

1.2.1.11. **Прикрепление и открепление задач (R-12).** Право R-12 позволяет прикреплять и откреплять задачи к проекту.

1.2.1.12. **Удаление проекта (R-13).** Право R-13 позволяет удалять проект.

1.2.1.13. **Управление автоматизациями проекта (R-11).** Право R-11 позволяет создавать, изменять, включать и отключать автоматизации проекта.

1.2.1.14. **Импорт задач в проект (R-14).** Право R-14 позволяет импортировать задачи в проект из внешнего формата.

#### 1.2.2 Модель доступов

1.2.2.1. **Унифицированная модель.** Все права в системе выражаются через набор R-прав (R-1…R-14) и применяются к задачам и проектам через прямые доступы, проектные доступы и доступы команд.

#### 1.2.3 Прямой доступ

1.2.3.1. **Определение прямого доступа.** Прямой доступ — доступ к конкретной задаче, выдаваемый пользователю или команде.

1.2.3.2. **Область действия прямого доступа.** Прямой доступ применяется только к той задаче, к которой он выдан.

1.2.3.3. **Набор прав прямого доступа.** Прямой доступ включает подмножество прав R-1…R-8.

#### 1.2.4 Проектный доступ

1.2.4.1. **Определение проектного доступа.** Проектный доступ — доступ, получаемый участником проекта (пользователем или командой).

1.2.4.2. **Область действия проектного доступа.** Проектный доступ распространяется на все задачи проекта.

1.2.4.3. **Набор прав проектного доступа.** Проектный доступ включает права R-1…R-14 в рамках контекста проекта.

#### 1.2.5 Видимость задач

1.2.5.1. **Чтение не является правом.** Чтение не является отдельным правом в системе.

1.2.5.2. **Видимость через проект.** Участник проекта видит все задачи проекта.

1.2.5.3. **Видимость через прямой доступ.** Наличие прямого доступа с любым правом даёт возможность чтения конкретной задачи.

1.2.5.4. **Невидимость без доступа.** Задача невидима для пользователя, не имеющего доступа к ней ни через проект, ни через прямой доступ.

#### 1.2.6 Вычисление итоговых прав

1.2.6.1. **Объединение источников доступа.** Итоговые права пользователя на задачу формируются как объединение (union) всех источников: прямых доступов к задаче, доступов команд пользователя, проектных доступов пользователя и проектных доступов команд пользователя.

1.2.6.2. **Отсутствие приоритета.** Ни один источник доступа не имеет приоритета над другим — все источники равнозначны.

#### 1.2.7 Шаринг и ограничения выдачи прав

1.2.7.1. **Шаринг задачи.** Право R-7 согласно п. 1.2.1.7 «Шаринг задачи» позволяет выдавать прямые доступы другим пользователям и командам.

1.2.7.2. **Ограничение на выдачу прав.** Пользователь не может выдать больше прав, чем имеет сам на момент выдачи. Ограничение распространяется на все механизмы выдачи прав, включая управление участниками проекта по праву R-10 согласно п. 1.2.1.10 «Управление участниками проекта».

#### 1.2.8 API доступов

1.2.8.1. **Выдача прямого доступа.** Система предоставляет API для выдачи прямого доступа (R-1…R-8) к задаче пользователю или команде через единый endpoint с полем subject_type: user|team.

1.2.8.2. **Обновление прямого доступа.** Система предоставляет API для изменения набора прав прямого доступа.

1.2.8.3. **Отзыв прямого доступа.** Система предоставляет API для отзыва прямого доступа.

1.2.8.4. **Список прямых доступов.** Система предоставляет API для получения списка прямых доступов задачи (пользователи и команды вместе, с полем subject_type: user|team).

1.5.1.1. **Authorization Code Flow.** Система использует OAuth 2.0 Authorization Code Flow как единый механизм аутентификации для всех клиентов. Клиент авторизации выступает authorization server.

1.5.1.2. **Регистрация.** Система предоставляет API для регистрации пользователя.

1.5.1.3. **Подтверждение email.** Система предоставляет API для подтверждения email пользователя.

1.5.1.4. **Авторизация (authorize).** Система предоставляет endpoint авторизации, принимающий client_id и redirect_uri. После успешного логина выполняется редирект на redirect_uri с authorization code.

1.5.1.5. **Обмен кода на токен.** Система предоставляет endpoint обмена authorization code на access token и refresh token.

1.5.1.6. **Обновление токена.** Система предоставляет endpoint обновления access token по refresh token.

1.5.1.7. **Логаут.** Система предоставляет API для выхода из системы с отзывом токенов.

#### 1.5.2 Клиент авторизации

1.5.2.3. **Подтверждение email.** Клиент авторизации обрабатывает переход по ссылке из письма, подтверждает email через API и автоматически перенаправляет пользователя в веб-приложение.

1.5.2.4. **Поддержка redirect_uri.** Клиент авторизации поддерживает redirect_uri на внешние адреса (веб-приложение) и на localhost (CLI).

#### 1.5.3 Удаление аккаунта

1.5.3.1. **Удаление прямых доступов при удалении пользователя.** При удалении пользователя удаляются все прямые доступы, выданные им другим пользователям и командам.

### 1.7 Тарифы и лимиты

#### 1.7.1 Тарифная система

1.7.1.1. **Привязка тарифа к пользователю.** Тариф привязан к пользователю и определяет доступные лимиты системы.

1.7.1.2. **Тарифный пул.** Тариф владельца распространяется на все созданные им проекты и команды, а также на сущности, созданные его managed-пользователями. Все ресурсы учитываются в рамках одного тарифного пула.

#### 1.7.2 Лимиты тарифов

1.7.2.1. **Лимит задач.** Free — до 200 задач. Pro, Team, Enterprise — неограниченно.

1.7.2.2. **Лимит проектов.** Free — до 3 проектов. Pro — до 20. Team, Enterprise — неограниченно.

1.7.2.3. **Лимит команд.** Free — до 3 команд. Pro — до 20. Team, Enterprise — неограниченно.

1.7.2.4. **Лимит автоматизаций.** Free — до 3 автоматизаций на проект. Pro — до 20 на проект. Team, Enterprise — неограниченно.

1.7.2.5. **Лимит участников.** Free — до 10 участников в проекте или команде. Pro — до 50. Team — до 100. Enterprise — 100 по умолчанию + слоты согласно п. 1.7.7 «Enterprise-слоты».

#### 1.7.3 Расчёт лимита задач

1.7.3.1. **Глобальный счётчик задач.** Лимит задач считается как единый глобальный счётчик задач пользователя и его managed-пользователей, независимо от проектов, статусов и состояния Freeze.

1.7.3.2. **Проектный механизм проверки.** При создании задачи или прикреплении к проекту проверяется лимит владельца проекта. При превышении лимита операция запрещается для всех пользователей проекта.

1.7.3.3. **Проверка при выдаче прямого доступа.** При выдаче прямого доступа проверяется лимит грантора (выдающего). Получатель доступа не влияет на лимит. В тарифах Pro, Team и Enterprise ограничение отсутствует.

#### 1.7.4 Заморозка

1.7.4.1. **Состояние Freeze.** При превышении лимитов или изменении тарифа часть сущностей переводится в состояние Freeze.

1.7.4.2. **Ограничения замороженной сущности.** Замороженная сущность доступна только для чтения; операции изменения, удаления и управления доступами блокируются.

1.7.4.3. **Автоматическое применение Freeze.** Freeze применяется автоматически на уровне системы и может затрагивать отдельные сущности или группы сущностей в рамках тарифного пула.

1.7.4.4. **Порядок заморозки.** При заморозке первыми замораживаются сущности с наиболее поздней датой создания (по created_at desc).

#### 1.7.5 Банк дней

1.7.5.1. **Механизм банка дней.** Неиспользованные оплаченные дни при апгрейде тарифа сохраняются в банк дней.

1.7.5.2. **Применение банка дней.** Банк применяется последовательно при завершении текущего периода тарифа.

#### 1.7.6 Изменение тарифа

1.7.6.1. **Апгрейд тарифа.** При апгрейде новые лимиты применяются сразу, неиспользованные дни предыдущего тарифа сохраняются в банк дней согласно п. 1.7.5.1 «Механизм банка дней».

1.7.6.2. **Даунгрейд тарифа.** При даунгрейде новый тариф вступает в силу после окончания оплаченного периода; возможна заморозка части сущностей согласно п. 1.7.4 «Заморозка».

#### 1.7.7 Enterprise-слоты

1.7.7.1. **Единица слота.** Один слот = +1 участник сверх базового лимита 100 участников.

1.7.7.2. **Область применения слотов.** Слоты применяются глобально ко всем проектам и командам владельца в рамках его тарифного пула.

1.7.7.3. **Масштабирование слотов.** Слоты масштабируются линейно без верхнего предела.

### 1.9 RSQL

#### 1.9.1 Общие положения

1.9.1.1. **Назначение RSQL.** RSQL используется для фильтрации списков задач. Диалект построен на стандартном RSQL/FIQL.

#### 1.9.2 Операторы сравнения

1.9.2.1. **Оператор равенства (==).** Оператор `==` проверяет равенство значения поля указанному значению.

1.9.2.2. **Оператор неравенства (!=).** Оператор `!=` проверяет неравенство значения поля указанному значению.

1.9.2.3. **Оператор «больше» (=gt=).** Оператор `=gt=` проверяет, что значение поля строго больше указанного.

1.9.2.4. **Оператор «меньше» (=lt=).** Оператор `=lt=` проверяет, что значение поля строго меньше указанного.

1.9.2.5. **Оператор «больше или равно» (=ge=).** Оператор `=ge=` проверяет, что значение поля больше или равно указанному.

1.9.2.6. **Оператор «меньше или равно» (=le=).** Оператор `=le=` проверяет, что значение поля меньше или равно указанному.

1.9.2.7. **Оператор вхождения (=in=).** Оператор `=in=` проверяет, что значение поля входит в указанный список: `status=in=(opened,progress)`.

1.9.2.8. **Оператор исключения (=out=).** Оператор `=out=` проверяет, что значение поля не входит в указанный список: `status=out=(closed)`.

#### 1.9.3 Логические операторы

1.9.3.1. **Оператор AND.** Точка с запятой (`;`) является оператором логического И.

1.9.3.2. **Оператор OR.** Запятая (`,`) является оператором логического ИЛИ.

1.9.3.3. **Группировка скобками.** Выражения группируются круглыми скобками: `(a;b),c`.

#### 1.9.4 Поля фильтрации

1.9.4.1. **Доступные поля.** Для фильтрации доступны все поля задачи: id, title, description, status, author, assignee, projects, tags, blocking_tasks, blocked_tasks, created_at, updated_at.

#### 1.9.5 Идентификация сущностей

1.9.5.1. **Идентификация пользователей.** Пользователи (author, assignee) идентифицируются по email: `author==ivanov@company.com`.

1.9.5.2. **Идентификация проектов.** Проекты (projects) идентифицируются по ID (UUID).

1.9.5.3. **Идентификация задач.** Задачи (blocking_tasks, blocked_tasks, id) идентифицируются по ID (UUID).

#### 1.9.6 Работа с массивами

1.9.6.1. **Поля-массивы.** Поля tags, projects, blocking_tasks, blocked_tasks являются массивами с перегруженными операторами.

1.9.6.2. **Оператор == для массивов.** Оператор `==` проверяет, что массив содержит указанный элемент: `tags==urgent`.

1.9.6.3. **Оператор != для массивов.** Оператор `!=` проверяет, что массив не содержит указанный элемент: `tags!=urgent`.

1.9.6.4. **Оператор =in= для массивов.** Оператор `=in=` проверяет, что пересечение массива с перечисленными значениями непусто: `tags=in=(urgent,critical)`.

1.9.6.5. **Оператор =out= для массивов.** Оператор `=out=` проверяет, что массив не содержит ни одного из перечисленных значений: `tags=out=(archived,stale)`.

#### 1.9.7 Enum-поля

1.9.7.1. **Сравнение статуса.** Статус сравнивается по значению из перечисления: `status==opened`.

#### 1.9.8 Пустые значения

1.9.8.1. **Литерал null.** Литерал `null` обозначает отсутствие значения.

1.9.8.2. **Проверка отсутствия значения.** Выражение `field==null` проверяет, что значение отсутствует (для assignee — не назначен, для массива — пуст).

1.9.8.3. **Проверка наличия значения.** Выражение `field!=null` проверяет, что значение присутствует (для assignee — назначен, для массива — непуст).

1.9.8.4. **Эквивалентность null и пустого массива.** Пустой массив и null эквивалентны на уровне семантики.

#### 1.9.9 Работа с датами

1.9.9.1. **Поля дат.** Поля created_at и updated_at поддерживают фильтрацию по дате.

1.9.9.2. **Формат ISO 8601.** Поддерживается формат ISO 8601 с таймзоной: `2025-01-15T10:00:00Z`, `2025-01-15T10:00:00+03:00`.

1.9.9.3. **Дата без времени.** Дата без времени (`2025-01-15`) трактуется как `2025-01-15T00:00:00Z`.

1.9.9.4. **Относительные даты.** Поддерживаются относительные даты: `now`, `now-30m`, `now-1h`, `now-7d`.

#### 1.9.10 Экранирование

1.9.10.1. **Экранирование строк.** Строковые значения со спецсимволами (пробелы, кавычки, запятые, скобки) оборачиваются в одинарные или двойные кавычки с экранированием внутренних кавычек обратным слэшем: `title=='Release "v2"'`.

#### 1.9.11 Регистр

1.9.11.1. **Регистр имён полей.** Имена полей регистрозависимы и записываются в нижнем регистре.

1.9.11.2. **Регистр значений.** Значения строковых сравнений регистрозависимы.

#### 1.9.12 Лимиты

1.9.12.1. **Максимальная длина выражения.** Максимальная длина строки RSQL-выражения — 4096 символов.

1.9.12.2. **Максимум операндов.** Максимум операндов в операторах `=in=` и `=out=` — 1000.

1.13.2.1. **Формат CSV.** Файл импорта имеет формат CSV с фиксированными именами колонок.

1.13.2.2. **Обязательная колонка title.** Колонка title является обязательной.

1.13.2.3. **Опциональная колонка description.** Колонка description является опциональной.

1.13.2.4. **Опциональная колонка status.** Колонка status опциональна; допустимые значения: opened, progress, closed. Пустая ячейка или отсутствие колонки → opened.

1.13.2.5. **Опциональная колонка assignee.** Колонка assignee опциональна; содержит email существующего пользователя. Если пользователь не найден — ошибка валидации.

1.13.2.6. **Опциональная колонка tags.** Колонка tags опциональна; теги в ячейке разделяются точкой с запятой: `urgent;backend;infra`.

1.13.2.7. **Опциональная колонка blocking_tasks.** Колонка blocking_tasks опциональна; содержит список ID существующих задач через точку с запятой. Если хотя бы один ID не найден — ошибка валидации.

1.13.2.8. **Валидация формата.** Отсутствие обязательной колонки title, неизвестная колонка, невалидное значение status, неизвестный email в assignee — ошибка валидации.

#### 1.13.3 Транзакционность

1.13.3.1. **Атомарность импорта.** Импорт выполняется по принципу all-or-nothing: если хотя бы одна строка невалидна или после импорта превышался бы лимит задач тарифа владельца проекта — ничего не импортируется.

1.13.3.2. **Отчёт об ошибках.** При ошибке валидации возвращается отчёт с ошибками по строкам.

#### 1.13.4 Ограничения файла

1.13.4.1. **Лимит строк.** Максимум 10 000 строк в файле импорта.

1.13.4.2. **Лимит размера.** Максимум 10 МБ размер файла импорта.

1.13.4.3. **Единый лимит для всех тарифов.** Ограничения на файл одинаковы для всех тарифов.

#### 1.13.5 API импорта

1.13.5.1. **Импорт задач в проект.** Система предоставляет API для импорта задач в проект из CSV-файла.

### 1.14 Массовые операции

#### 1.14.1 Режимы выполнения

1.14.1.1. **Режим Bulk.** Bulk — атомарное выполнение: операция применяется полностью или не применяется вовсе.

1.14.1.2. **Режим Batch.** Batch — частичное выполнение: система применяет изменения к каждой задаче независимо и возвращает детальный результат по каждой.

#### 1.14.2 Целевое множество

1.14.2.2. **Выбор по списку ID.** Целевое множество задач может быть задано явным списком идентификаторов.

#### 1.14.3 Изменяемые поля

#### 1.14.4 Проверка прав

1.14.4.1. **Проверка прав per-задача.** Права проверяются для каждой задачи индивидуально на каждое изменяемое поле.

#### 1.14.5 API массовых операций

1.14.5.1. **Bulk по RSQL.** Система предоставляет API для массового изменения задач по RSQL-выражению в режиме bulk.

1.14.5.2. **Batch по RSQL.** Система предоставляет API для массового изменения задач по RSQL-выражению в режиме batch.

## 2. Нефункциональные требования

### 2.1 Безопасность транспорта

2.1.1. **TLS termination.** Все внешние подключения к API обслуживаются через HTTPS. TLS termination выполняется на уровне API Gateway; внутренний трафик между сервисами передаётся по защищённой сети кластера.

2.1.2. **Идентификатор запроса.** Каждый входящий запрос к API получает уникальный request_id, который передаётся в заголовках ответа и прокидывается через все внутренние вызовы для сквозной трассировки.

### 2.2 Ограничение частоты запросов

2.2.1. **Rate limiting по IP.** API Gateway ограничивает количество запросов с одного IP-адреса в единицу времени. При превышении лимита возвращается HTTP 429 Too Many Requests.

2.2.2. **Rate limiting по токену.** API Gateway ограничивает количество запросов с одного PAT-токена в единицу времени. При превышении лимита возвращается HTTP 429 Too Many Requests.
